## 1. Executive Summary

The **Custom Domain Management** system allows UBOTZ 2.0 tenants to replace their platform-default subdomain (e.g., `tenant.ubotz.io`) with a private, branded domain (e.g., `academy.tenant.com`). 

This is a premium feature governed by subscription tiers. The system enforces a rigorous multi-step DNS verification process to ensure the tenant actually owns the domain and has correctly configured their DNS records before the domain is activated for public traffic.

---

## 2. Domain Lifecycle & States

Every custom domain follows a governed lifecycle. A tenant can only have **one active or pending custom domain** at any time.

| Status | DB Value | Meaning |
|---|---|---|
| `PENDING_VERIFICATION` | `pending_verification` | Domain added but DNS not yet verified. |
| `ACTIVE` | `active` | DNS verified. Public traffic is now routed to this domain. |
| `VERIFICATION_FAILED` | `verification_failed` | 72-hour deadline passed without successful DNS validation. |
| `INACTIVE_DNS_FAILURE` | `inactive_dns_failure` | Was active, but a routine DNS check failed. |
| `INACTIVE_PLAN_DOWNGRADE` | `inactive_plan_downgrade` | Deactivated because the tenant's new plan doesn't support custom domains. |
| `REMOVED` | `removed` | Soft-deleted. The domain record is a terminal state — **cannot transition further**. The domain is released for other tenants after a configurable cooldown. |

> A tenant can only have **one active or pending custom domain** at any time. Once removed, a brand-new domain record must be created to add a domain again.

---

## 3. The Verification Process

To activate a domain, the tenant must complete two verification checks:

1. **CNAME Verification**: The domain (or subdomain) must be pointed to the platform's central load balancer target (e.g., `proxy.ubotz.io`).
2. **TXT Verification**: The tenant must add a specific TXT record containing a unique `verification_token` generated by the platform.

**Verification Deadline**: Tenants have **72 hours** from the moment they add the domain to complete verification. If the deadline passes without both records being correctly detected, the domain moves to `VERIFICATION_FAILED`.

---

## 4. Automated Enforcement

The system performs ongoing checks to ensure domain integrity:

- **DNS Monitoring**: Routine background checks verify that CNAME and TXT records remain intact. If they are removed, the domain is flagged for `dns_failure` (stored as `dns_verification_failed`) and eventually deactivated.
- **Plan Enforcement**: If a tenant downgrades to a plan that excludes the `custom_domain` feature, the domain is immediately transitioned to `INACTIVE_PLAN_DOWNGRADE` from **any non-REMOVED state** (including `PENDING_VERIFICATION`).
- **Cooldown Period**: When a domain is removed, it may enter a cooldown phase where it cannot be claimed by other tenants immediately, preventing domain-squatting or accidental takeovers (`DomainCooldownException`).
- **Manual Verify Rate Limiting**: Tenants who trigger verification checks too rapidly receive a `429` rate-limit response (`ManualVerifyRateLimitException`).
- **Reverse-Proxy Cache**: An active domain's tenant mapping is cached in Redis (`custom_domain:{domain}` key, 5-min TTL) to serve Caddy routing decisions without DB hits. The cache is invalidated immediately on deactivation or removal.

## 5. Business Rules — Complete Reference

| ID | Rule | Enforcement Point |
|---|---|---|
| BR-DOM-01 | Only tenants with the `custom_domain` feature enabled in their subscription plan can add a domain. | `AddCustomDomainUseCase` — checks plan features. |
| BR-DOM-02 | A tenant can only have one custom domain (active or pending) at a time. | `AddCustomDomainUseCase` — checks `findByTenantId`. |
| BR-DOM-03 | Custom domains must be globally unique across the entire platform. | `AddCustomDomainUseCase` — checks `isDomainAvailable`. |
| BR-DOM-04 | Platform-owned domains (`*.ubotz.io`, `*.ubotz.com`) cannot be added as custom domains. | `TenantCustomDomain::validateDomainFormat`. |
| BR-DOM-05 | Activation requires BOTH CNAME and TXT records to be correctly resolved. | `VerifyCustomDomainDnsUseCase`, `activate()`, `reactivate()`. |
| BR-DOM-06 | Verification must be completed within 72 hours of addition. | `VerifyCustomDomainDnsUseCase` — checks deadline. |
| BR-DOM-07 | Removing a domain releases it for other tenants (after cooldown). | `RemoveCustomDomainUseCase`. |
| BR-DOM-08 | A plan downgrade that removes the `custom_domain` feature immediately deactivates any non-removed domain (including `PENDING_VERIFICATION`). | `ProcessCustomDomainDeactivationListener`. |
| BR-DOM-09 | Tenants are rate-limited on manual DNS verification requests to prevent abuse. | `VerifyCustomDomainDnsUseCase` — raises `ManualVerifyRateLimitException` (429). |

## 6. Security Guards

1. **Token Uniqueness**: Each domain registration generates a cryptographically secure, unique verification token (32-char hex via `bin2hex(random_bytes(16))`).
2. **Format Guard**: IP addresses, internal hostnames (`localhost`, `*.local`), platform subdomains (`*.ubotz.io`, `*.ubotz.com`), and malformed domains are rejected at the entity level (`InvalidDomainException`).
3. **Auditability**: All domain lifecycle changes (Added, Verified, Activated, Deactivated, Removed) are captured in the tenant's audit log with actor ID, timestamp, and metadata.
4. **Cooldown Protection**: A removed domain enters cooldown to prevent domain enumeration or squatting by other tenants (`DomainCooldownException`).
5. **Rate Limiting**: Manual DNS verify calls are rate-limited per tenant to prevent infrastructure abuse.
